# 超市独立版改造：H5 嵌套与服务商清理计划

## Summary

- 目标：基于当前“服务商多商家”源码，改造成“单超市主体 + 多门店”的独立版本。
- 前端形态：
  - `miniprogram` 继续作为主前端源码，但业务主体转为 `H5` 构建。
  - 用户端通过“小程序登录成功后携带 token 进入 H5”的方式无感进入。
  - 商家端不再嵌入小程序，改为直接访问 H5，并且只保留账号密码登录。
- 平台语义：
  - 删除服务商、多商家、分账、微信支付服务商模式、小程序品牌配置、七牛上传相关能力。
  - 统一将“服务商后台”改造成“总部后台 / admin”，将当前 `merchant` 的业务文案改造成“门店 / store”语义。
- 交付顺序：
  1. 后端上传改为本地存储并补齐静态访问。
  2. 后端删除微信支付、退款回调、分账、服务商路由与模型依赖。
  3. `web-admin` 改造成单超市多门店后台，接口前缀从 `/api/v1/sp/*` 切到 `/api/v1/admin/*`，部署基路径从 `/sp/` 切到 `/admin/`。
  4. `miniprogram` 调整为“H5 主体 + 小程序用户登录壳”的双形态。
  5. 以 H5 构建结果进行联调和测试。

## Current State Analysis

### 1. 前端现状

- `miniprogram/package.json`
  - 已同时支持 `mp-weixin` 与 `h5` 构建，适合直接演进为“H5 主体 + 小程序壳”。
- `miniprogram/src/pages.json`
  - 当前同时包含用户端页面和商家端页面，商家端路由集中在 `pages/merchant/*`，用户端在 `pages/store/*`。
- `miniprogram/src/pages/auth/login.vue`
  - 当前是商家登录页，支持账号密码和微信快捷登录。
- `miniprogram/src/utils/useAuth.ts`
  - 当前用户端通过 `uni.login + /api/v1/auth/user/wechat-login` 登录并持久化 `user_token`。
- `web-admin/src/*`
  - 当前完整服务商后台已独立成 Vite 项目，接口全部依赖 `/api/v1/sp/*`，页面文案和路由均为“服务商”语义。

### 2. 上传现状

- `server/pkg/qiniu/qiniu.go`
  - 当前上传体系完全围绕七牛实现，负责上传 token、域名与私有地址拼接。
- `server/internal/handlers/upload/upload.go`
  - 当前只提供 `GET /api/v1/upload/token`，供前端直传七牛。
- `miniprogram/src/api/index.ts`
  - 当前 `uploadImage()` 基于上传凭证直传七牛。
- `web-admin/src/utils/qiniu.ts`
  - 当前后台图片上传也是先取 token 再直传七牛。

### 3. 支付与分账现状

- `server/internal/services/wechatpay/service_provider.go`
  - 当前封装的是微信服务商支付、退款、分账能力。
- `server/internal/handlers/sp/payment.go`
  - 当前承担支付回调、退款回调、分账落账。
- `server/internal/handlers/user/handler.go`
  - 当前创建订单时会生成 `pay_params` 并走微信支付下单。
- `miniprogram/src/pages/store/confirm.vue`
  - 当前下单完成后直接调用 `uni.requestPayment`。
- `web-admin/src/views/settlement/ProfitSharingHistoryView.vue`
  - 当前后台含完整分账历史页面。

### 4. 服务商体系现状

- `server/cmd/server/main.go`
  - 当前存在完整 `/api/v1/sp/*` 路由组，以及微信支付回调路由。
- `server/internal/models/models.go`
  - 当前存在 `service_providers`、`service_provider_sps`，且 `merchants`、`announcements`、`activities`、`merchant_profit_sharing_records` 等均带 `service_provider_id`。
- `web-admin/src/api/sp.ts`
  - 当前后台 API 全量绑定 `/api/v1/sp/*`。
- `web-admin/src/router/index.ts` 与 `web-admin/vite.config.ts`
  - 当前前端路由与部署路径都绑定 `/sp`。

## Assumptions & Decisions

- 数据建模决策：
  - 本轮改造后的业务模型为“单超市主体 + 多门店”。
  - 数据库层首轮不做 `merchants` 到 `stores` 的物理表重命名，保留 `merchants` 表承载“门店实体”，避免一次性大迁移。
  - 语义层统一改为“门店 / store”，后续若需要再单独做表名重构。
- 前端决策：
  - H5 主体继续基于 `miniprogram` 演进，不新开用户端项目。
  - 商家端也继续复用 `miniprogram` 中的门店端页面做 H5，但不再通过小程序承载。
  - `web-admin` 保留并改造成总部后台，用于总部维护门店、订单、公告、活动等管理能力。
- 登录决策：
  - 用户端：小程序先登录，成功后通过 URL 参数直接把 token 带入 H5。
  - 商家端：删除微信快捷登录，仅保留账号密码登录。
  - 小程序壳仅服务用户端，不承载商家端。
- 命名决策：
  - 服务商命名本轮一起清理。
  - 后端接口前缀统一改为 `/api/v1/admin/*`。
  - `web-admin` 部署基路径统一改为 `/admin/`。
- 支付决策：
  - 本轮彻底移除微信支付、退款、分账、服务商支付配置相关逻辑。
  - 下单链路先改为“创建订单成功但不发起在线支付”的待接入状态，为后续江苏银行微信支付接入预留干净基线。
- 文档决策：
  - 因涉及产品流程、接口和数据库字段调整，必须同步更新 PRD 文档。

## Proposed Changes

### A. 后端：本地文件存储替代七牛

#### 目标

- 删除七牛依赖，改为服务端接收文件并保存到配置指定目录。
- 为前端提供稳定可访问的静态资源 URL。

#### 文件与改法

- `server/internal/config/config.go`
  - 删除 `Qiniu` 配置结构和相关加载逻辑。
  - 新增本地存储配置，例如：
    - `UPLOAD_ROOT_DIR`
    - `UPLOAD_PUBLIC_BASE_URL`
    - `UPLOAD_URL_PREFIX`
  - 目的：把“落盘目录”和“外部访问地址”从代码中解耦。

- `server/.env.example`
  - 删除七牛相关示例变量。
  - 增加本地存储配置示例。

- `server/cmd/server/main.go`
  - 删除 `pkg/qiniu` 初始化。
  - 增加静态文件服务挂载，把上传目录映射到统一访问前缀，例如 `/uploads/*`。
  - 保留访问路径与磁盘路径解耦，避免前端感知物理目录。

- `server/internal/handlers/upload/upload.go`
  - 删除 `GET /api/v1/upload/token` 的七牛凭证返回逻辑。
  - 改为受鉴权保护的服务端上传接口，建议统一为 `POST /api/v1/upload/file`，使用 `multipart/form-data`。
  - 响应改为直接返回：
    - `path`：数据库存储的相对路径
    - `url`：前端可直接展示的访问地址
  - 上传目录规则：
    - 公共上传：`common/yyyy/mm/dd`
    - 门店上传：`store/{merchant_id}/yyyy/mm/dd`
    - 后台上传：`admin/yyyy/mm/dd`

- 新增 `server/internal/storage/local.go` 或 `server/pkg/storage/local.go`
  - 封装保存文件、生成相对路径、拼接公开 URL 的逻辑。
  - 替代现有 `pkg/qiniu` 的“存储入口”职责。

- `server/internal/handlers/user/handler.go`
- `server/internal/handlers/sp/handler.go`
- `server/internal/services/orderquery/query.go`
  - 把当前 `BuildPrivateURL()` 之类的七牛地址拼接逻辑改为本地 URL 构造函数。
  - 统一兼容数据库里已经存在的相对路径。

- `miniprogram/src/api/index.ts`
- `miniprogram/src/api/qiniu_upload.ts`
  - 删除七牛 token 获取与直传实现。
  - 改为调用新的服务端文件上传接口。
  - `qiniu_upload.ts` 直接删掉或并入统一上传模块。

- `miniprogram/src/pages/merchant/products/edit.vue`
  - 上传商品图时改用新的服务端上传接口。

- `web-admin/src/utils/qiniu.ts`
- `web-admin/src/api/sp.ts`
- `web-admin/src/views/merchant/MerchantDetailView.vue`
  - 删除七牛工具和上传 token API。
  - 改为浏览器直传到后端 `multipart/form-data` 上传接口。

## Risks

- 用户端“URL 直接带 token”方案实现最快，但安全性弱于短时票据；本轮按你的决策执行，后续如上线公网建议升级为一次性票据换 token。
- 由于本轮同时涉及接口、路由、上传、支付、数据库字段和文案，执行时必须按顺序逐步提交并频繁验证，避免大面积回归。
- `merchant` 数据表继续承载“门店”语义，能降低首轮改造风险，但代码里会存在“表名不改、文案先改”的过渡期，需要在文档中明确。
