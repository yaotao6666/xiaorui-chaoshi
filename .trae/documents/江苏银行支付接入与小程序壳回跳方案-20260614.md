# 江苏银行支付接入与小程序壳回跳方案

## Summary

目标是在现有“`xcx` 壳登录 + `WebView` 打开 H5 + Go 后端创建订单”基线上，接入 `api-chaoshi/certs/` 下的江苏银行真实支付参数与证书能力，先跑通“小程序支付”主链路，并为后续 PC 后台扫码支付预留扩展位。

本轮范围与关键决策已固定：

- 只落地“小程序支付”主链路，扫码支付本轮只预留后台入口能力设计，不实现前后端完整交付。
- 订单与支付拆分：保留现有“创建订单”接口负责落单，再新增“发起支付”接口。
- H5 在 `xcx` 壳内创建订单成功后，回到 `xcx` 的新支付页，由该页调用 `Taro.requestPayment`。
- 支付完成后默认返回 H5 订单详情页。
- 普通浏览器打开的 H5 允许创建待支付订单，但只提示“请通过小程序完成支付”，不直接发起支付。

## Proposed Changes

### 1. 后端支付配置与证书接入

#### 修改文件

- `e:\yt\xiaorui-chaoshi\api-chaoshi\internal\config\config.go`
- `e:\yt\xiaorui-chaoshi\api-chaoshi\.env.example`
- `e:\yt\xiaorui-chaoshi\api-chaoshi\.env.production.example`

#### 变更内容

- 新增独立的 `JsBankPay` 配置结构，不复用当前偏“原生微信支付服务商”命名的 `WechatPay`。
- 配置项按江苏银行样例落成，建议统一为：
  - `JSBANK_PAY_ENABLED`
  - `JSBANK_PAY_APP_ID`
  - `JSBANK_PAY_MCH_ID`
  - `JSBANK_PAY_MCH_NAME`
  - `JSBANK_PAY_MASTER_ACCOUNT`
  - `JSBANK_PAY_PARTNER_ID`
  - `JSBANK_PAY_DEVICE_NO`
  - `JSBANK_PAY_POST_URL`
  - `JSBANK_PAY_NOTIFY_URL`
  - `JSBANK_PAY_PFX_PATH`
  - `JSBANK_PAY_PFX_PASSWORD`
  - `JSBANK_PAY_PUBLIC_CERT_PATH`
- `JSBANK_PAY_APP_ID` 默认允许与 `WECHAT_APP_ID` 相同；若未单独配置，则回退到 `WECHAT_APP_ID`。

#### 设计原因

- 江苏银行这套能力与当前 `WechatPay` 配置结构语义不一致，单独建模更清晰，也能避免后续误把 APIv3 等旧字段带入新链路。

### 2. 新增 Go 版江苏银行支付服务

#### 新增文件

- `e:\yt\xiaorui-chaoshi\api-chaoshi\internal\services\payment\jsbank\client.go`
- `e:\yt\xiaorui-chaoshi\api-chaoshi\internal\services\payment\jsbank\signer.go`
- `e:\yt\xiaorui-chaoshi\api-chaoshi\internal\services\payment\jsbank\types.go`

#### 变更内容

- 用 Go 复刻 `JsPayService.php` 的核心能力：
  - 加载 `.pfx` 私钥
  - 读取 `.cer` 公钥证书
  - 参数排序与签名
  - 发送江苏银行表单请求
  - 解析 `k=v&...` 响应报文
  - 验签异步通知
- `golang.org/x/crypto/pkcs12` 已在 `go.mod` 中存在，可直接用来解析 PFX。
- 对外暴露最小必要方法：
  - `CreateMiniProgramPay(req)`：预下单，返回小程序支付参数
  - `VerifyNotify(params)`：验证回调

#### 设计原因

- 这层要独立于 `handler`，避免支付参数构造、签名、HTTP 请求和业务状态流转混在控制器里。

### 3. 新增支付发起与回调接口

#### 修改文件

- `e:\yt\xiaorui-chaoshi\api-chaoshi\cmd\server\main.go`
- `e:\yt\xiaorui-chaoshi\api-chaoshi\internal\handlers\user\handler.go`

#### 新增接口

1. `POST /api/v1/user/orders/:order_id/pay/prepare`
2. `POST /api/v1/payments/jsbank/notify`

#### `POST /api/v1/user/orders/:order_id/pay/prepare` 设计

- 鉴权：`JWTAuth + UserAuth`
- 入参：
  - `return_path`：H5 支付完成后要回到的 hash 路径，默认 `/pages/store/order-detail`
  - `source`：调用来源，固定透传 `xcx_shell`
- 核心逻辑：
  - 校验订单归属当前用户
  - 校验订单状态必须是 `1=待支付`
  - 校验订单 `pay_amount > 0`
  - 通过订单上的用户 ID 查询 `openid`
  - 调用 `jsbank.CreateMiniProgramPay`
  - 组装并返回给 `xcx` 的支付参数
- 返回结构固定为：

```json
{
  "order_id": 123,
  "order_no": "202606140001",
  "merchant_id": 2,
  "pay_amount": 18.6,
  "channel": "jsbank_wechat_mini",
  "pay_params": {
    "timeStamp": "1710000000",
    "nonceStr": "xxxxx",
    "package": "prepay_id=xxxxx",
    "signType": "RSA",
    "paySign": "xxxxx"
  },
  "return_target": "/pages/store/order-detail?order_id=123&merchant_id=2"
}
```

#### `POST /api/v1/payments/jsbank/notify` 设计

- 不鉴权，按江苏银行回调协议接收表单参数。
- 核心逻辑：
  - 使用 `.cer` 验签
  - 根据银行报文中的商户订单号映射到 `orders.order_no`
  - 幂等更新订单：
    - `status`：`1 -> 2`
    - `transaction_id`
    - `paid_at`
    - `pay_notify_payload`
  - 同步更新用户 `has_paid/first_paid_at`
- 返回按江苏银行要求的成功应答字符串；若验签失败或处理失败，记录日志并返回失败响应。

#### 设计原因

- 创建订单与发起支付解耦，可以保留当前下单主流程，降低对现有结算逻辑的冲击。
- 回调单独路由，便于网关、Nginx 和银行后台单独配置。

### 4. 调整订单创建接口返回，加入“下一步动作”而非直接发起支付

#### 修改文件

- `e:\yt\xiaorui-chaoshi\api-chaoshi\internal\handlers\user\handler.go`
- `e:\yt\xiaorui-chaoshi\docs\prd\PRD-接口文档.md`

#### 变更内容

- 保留 `CreateOrder()` 当前的落单逻辑与订单状态流转。
- 将返回结构从“纯 pending_integration 占位”调整为“明确下一步动作”：
  - `enabled`
  - `status`
  - `message`
  - `next_action`
  - `prepare_url`
- 行为约定：
  - `pay_amount == 0`：直接视为“无需支付”，前端跳订单详情。
  - `pay_amount > 0` 且来源是 `xcx_shell`：返回 `next_action=open_xcx_payment`
  - `pay_amount > 0` 且不是 `xcx_shell`：返回 `next_action=show_xcx_pay_guide`

#### 设计原因

- 创建订单接口不直接承担支付签名，避免 H5 在非壳环境也误拿到支付参数。

### 5. H5 增加“跳回小程序支付页”桥接能力

#### 修改文件

#### 变更内容

- `confirm.vue`
  - 创建订单成功后不再一律跳订单列表。
  - 若 `pay_amount > 0` 且当前来源是 `xcx_shell`，调用 H5 桥接方法跳转到 `xcx` 新支付页。
  - 若 `pay_amount > 0` 但不是 `xcx_shell`，跳订单详情并显示“请在小程序中完成支付”提示。
  - 若 `pay_amount == 0`，直接跳订单详情。
- `order-detail.vue`
  - 对 `status = 待支付` 的订单新增“去小程序支付”按钮。
  - 在 `xcx_shell` 场景下，该按钮直接跳 `xcx` 支付页。
  - 在普通 H5 场景下，按钮显示为引导说明，不调用支付。
- `miniProgramBridge.ts`
  - 封装：
    - 当前是否运行在微信环境
    - 当前是否由 `xcx_shell` 打开
    - 调用 `wx.miniProgram.navigateTo` 跳转 `xcx` 支付页
  - 使用微信 JSSDK `wx.miniProgram`，不引入复杂桥接协议。

#### 设计原因

- 当前 H5 在 `WebView` 里，无法直接调用 `Taro.requestPayment`；必须跳回 `xcx` 原生页面再发起小程序支付。

#### 变更内容

- 支付页入参：
  - `orderId`
  - `merchantId`
  - `returnTarget`
- 页面流程固定为：
  1. 进入页面后调用 `/api/v1/user/orders/:order_id/pay/prepare`
  2. 拿到 `pay_params`
  3. 调用 `Taro.requestPayment`
  4. 支付成功后，把 H5 订单详情地址重新拼成 `target`
  5. `redirectTo('/pages/webview/index?target=...')`
  6. 支付取消或失败时停留在当前页，可重试或返回 H5 订单详情
- 返回 H5 的目标页面固定为：
  - `/pages/store/order-detail?order_id={id}&merchant_id={merchant_id}`

#### 设计原因

- 把支付动作放在独立原生页，能保证：
  - 支付参数只在原生页使用
  - 成功/失败/取消都有统一承接页
  - 后续若加扫码支付结果承接，也有统一支付中心页面可复用

## Assumptions & Decisions

- 江苏银行这 4 个文件为真实接入资料，本轮直接作为实现依据，不再继续沿用 PHP 运行时。
- 订单主键、订单号和现有 `orders` 表支付字段足以支撑本轮主链路，不新增数据库表。
- 小程序支付只依赖当前用户 `openid`，该字段沿用现有微信登录体系。
- H5 只负责下单和展示，不直接持有原生支付动作；原生支付只在 `xcx` 新页面中触发。
- 本轮不实现：
  - 扫码支付前端页面
  - 退款接口与退款回调
  - 主动支付查询独立接口
  - 支付结果专用 H5 页面
- 对于已创建但未支付的订单，订单详情页承担“继续支付/引导去小程序支付”的入口。
