# H5在小程序内重复Title处理方案

## 一、Summary

目标是解决“`xcx` 小程序壳内打开 H5 时出现重复 title”的问题，并按已确认偏好落地为：

- 保留小程序外层原生导航栏标题
- 隐藏 H5 页面顶部头部
- 对整个壳内 H5 链路统一处理
- 小程序外层标题按 H5 当前路由动态切换

本方案只做标题与头部体验收口，不改变现有登录、下单、支付、回跳链路。

## 二、Current State Analysis

### 2.1 小程序壳当前实现

- `xcx` 入口页会拼装 H5 地址后进入 `pages/webview/index`
  - 参考文件：`xcx/src/pages/index/index.tsx`
- `web-view` 容器页当前仅渲染 `<WebView src={targetUrl} />`
  - 参考文件：`xcx/src/pages/webview/index.tsx`
- `web-view` 页当前原生标题固定为 `H5 容器`
  - 参考文件：`xcx/src/pages/webview/index.config.ts`
- 全局默认导航栏标题为 `超市 H5 验证壳`
  - 参考文件：`xcx/src/app.config.ts`

结论：

- 当前小程序外层原生导航栏一定会显示
- 标题是固定文案，不能跟随 H5 路由变化
- 当前 `xcx` 仓库里没有现成的 `web-view onMessage` 接收逻辑

### 2.2 H5 当前实现

- 当前小程序壳入口默认指向店铺 H5 首页：
  - `xcx/src/services/config.ts` 中 `h5EntryUrl = http://localhost:3000/#/pages/store/home`
- H5 内已存在小程序桥接工具，但当前只封装了 `navigateTo`
  - 参考文件：`miniprogram/src/utils/miniProgramBridge.ts`
- H5 中未发现现成的“小程序标题同步”消息通道
- H5 当前有明确的顶部视觉头部：
  - `miniprogram/src/pages/store/home.vue`
    - `store-header`
    - `sticky-mini-bar`
- 其余店铺页面（`confirm/cart/my-orders/order-detail/product/address-*`）未发现同等显式页面头部块，更多依赖页面内容区自身结构

结论：

- “重复 title”当前最明显的来源是：
  - 小程序原生导航栏
  - `store/home.vue` 的 H5 自定义头部/吸顶头部
- 若要实现“全部壳内 H5 页面统一体验”，除了隐藏显式头部，还需要让原生标题跟随 H5 路由同步

### 2.3 当前可利用的现有基础

- H5 已能识别是否处于小程序 `web-view`
  - `miniprogram/src/utils/miniProgramBridge.ts` 中已有 `isMiniProgramWebview()`
- H5 请求层已能从 URL 中恢复：
  - `token`
  - `openid`
  - `source`
  - 参考文件：`miniprogram/src/utils/request.ts`
- 店铺页普遍通过 `onLoad/getCurrentPages()` 读取路由参数
  - 参考文件：
    - `miniprogram/src/pages/store/home.vue`
    - `miniprogram/src/pages/store/confirm.vue`
    - `miniprogram/src/pages/store/product.vue`
    - `miniprogram/src/pages/store/cart.vue`
    - `miniprogram/src/pages/store/my-orders.vue`
    - `miniprogram/src/pages/store/order-detail.vue`

## 三、Assumptions & Decisions

### 3.1 已确认决策

- 标题策略：保留小程序外层原生标题，隐藏 H5 顶部头部
- 处理范围：壳内 H5 整体统一处理
- 标题文案：按路由动态标题

### 3.2 方案边界

- 当前壳入口只打开店铺 H5 路径，因此本次范围以 `miniprogram/src/pages/store/*` 为主
- 商家后台 H5 页面不作为本次壳内标题治理的主范围；除非后续确认也会被放进 `web-view`
- 本次不改支付页 `xcx/src/pages/payment/index.tsx`，它本身就是小程序原生页，继续保留原标题

### 3.3 技术决策

- 使用“小程序原生导航栏 + H5 路由标题映射”的组合方案
- 使用“H5 -> 微信 JSSDK -> 小程序 `web-view` message”建立轻量标题同步桥
- 使用“壳内上下文标记”控制 H5 页面是否隐藏头部
- 先覆盖当前明确存在重复头部的店铺首页与吸顶头部；其他店铺页统一接入标题同步能力，并保留后续扩展点

## 四、Proposed Changes

### 4.1 `xcx/src/pages/webview/index.tsx`

变更目的：

- 让小程序壳接收 H5 发来的标题消息
- 根据消息动态调用小程序导航栏标题更新
- 在首次加载时根据目标 URL 先做一次默认标题初始化，避免白屏期间仍显示 `H5 容器`

具体做法：

- 为 `<WebView />` 增加 `onMessage` 处理
- 解析消息体，仅接受约定格式，例如：
  - `type = "page_title_sync"`
  - `title = "店铺首页"`
- 收到有效消息后调用 `Taro.setNavigationBarTitle({ title })`
- 在 `useLoad` 中解析 `targetUrl` 的 hash 路径，按预定义映射先设置一次默认标题
- 兜底标题可使用：
  - `店铺首页`
  - `确认订单`
  - `商品详情`
  - `购物车`
  - `我的订单`
  - `订单详情`
  - `收货地址`
  - `编辑地址`
  - 未命中时回退到 `商家助手`

### 4.2 `xcx/src/pages/webview/index.config.ts`

变更目的：

- 避免继续使用调试性质的固定标题 `H5 容器`

具体做法：

- 将初始标题从 `H5 容器` 调整为更通用的默认值，例如 `商家助手`
- 实际展示标题以后续 `setNavigationBarTitle` 动态覆盖为准

### 4.3 `xcx/src/pages/webview/*` 或新增壳侧标题工具文件

变更目的：

- 把“hash 路径 -> 原生标题”的映射收敛成可维护逻辑

建议实现：

- 新增例如：
  - `xcx/src/utils/webviewTitle.ts`
- 统一提供：
  - 路径规范化
  - 标题映射函数
  - 默认标题回退

原因：

- 避免标题映射硬编码在页面组件里
- 后续新增 `store/*` 页面时只改一处

### 4.4 `miniprogram/src/utils/miniProgramBridge.ts`

变更目的：

- 在现有 `navigateTo` 能力外，补充“向小程序壳同步标题”的桥接能力

具体做法：

- 保留现有：
  - `isMiniProgramWebview()`
  - `openXcxPaymentPage()`
- 新增例如：
  - `syncMiniProgramTitle(title: string): Promise<void>`
- 实现方式：
  - 复用现有 JSSDK 加载逻辑
  - 调用微信小程序 `postMessage` 能力向外层 `web-view` 发送消息
- 增加防抖/去重：
  - 相同标题不重复发送

### 4.5 `miniprogram/src/App.vue` 或新增 H5 侧 composable / util

变更目的：

- 在 H5 全局层判断当前是否为 `xcx_shell`
- 在路由变化时统一同步标题

建议实现：

- 新增例如：
  - `miniprogram/src/utils/embeddedShell.ts`
  - 或 `miniprogram/src/composables/useEmbeddedShell.ts`
- 统一提供：
  - `isXcxShellSource()`
  - `getStorePageTitleByRoute()`
  - `syncCurrentPageTitle()`

实现要点：

- 根据 URL / hash / query 中的 `source=xcx_shell` 判断当前是否壳内
- 统一抽取店铺页路径到标题映射：
  - `/pages/store/home` -> `店铺首页`
  - `/pages/store/product` -> `商品详情`
  - `/pages/store/confirm` -> `确认订单`
  - `/pages/store/cart` -> `购物车`
  - `/pages/store/my-orders` -> `我的订单`
  - `/pages/store/order-detail` -> `订单详情`
  - `/pages/store/address-list` -> `收货地址`
  - `/pages/store/address-edit` -> `编辑地址`
- 在应用初始化与路由切换时触发标题同步

说明：

- 当前项目为 `uni-app + vue3`
- 若全局路由监听接入成本偏高，则退一步在 `store/*` 页面各自 `onLoad/onShow` 调一次共享标题同步函数
- 计划优先采用“共享函数 + 各页面显式调用”的方式，避免引入不稳定的全局 H5 路由监听改造

### 4.6 `miniprogram/src/pages/store/home.vue`

变更目的：

- 解决当前最明显的重复头部问题

具体做法：

- 在壳内场景下隐藏以下区域：
  - `store-header`
  - `sticky-mini-bar`
- 非壳内场景保持现有展示不变

实现方式：

- 引入共享的 `isXcxShell` 判断
- 模板层使用 `v-if` / `v-show` 控制头部显示
- 若头部隐藏后顶部留白异常，同时补齐相应容器间距调整

### 4.7 `miniprogram/src/pages/store/*.vue`

建议覆盖页面：

- `pages/store/home.vue`
- `pages/store/product.vue`
- `pages/store/confirm.vue`
- `pages/store/cart.vue`
- `pages/store/my-orders.vue`
- `pages/store/order-detail.vue`
- `pages/store/address-list.vue`
- `pages/store/address-edit.vue`

变更目的：

- 让所有壳内店铺 H5 页面在进入/返回时都能刷新小程序外层标题

具体做法：

- 每个页面在 `onLoad` 或 `onShow` 中调用共享标题同步函数
- 只在 `source=xcx_shell` 时执行
- 页面本身没有显式顶部头部的，不做视觉结构调整，只做标题同步

### 4.8 `miniprogram/src/pages/store/confirm.vue` 与 `order-detail.vue`

变更目的：

- 保障支付回跳后的标题仍正确

原因：

- 当前支付链路存在：
  - H5 下单
  - 跳回 `xcx` 原生支付页
  - 支付完成后重新打开 H5 订单详情
- 这些页面回流频繁，必须确保回跳后原生标题同步正确

具体做法：

- 在进入页面和支付回跳恢复时都调用标题同步函数

## 五、Implementation Steps

1. 在 `xcx` 侧抽取 `web-view` 路由标题映射工具
2. 改造 `xcx/src/pages/webview/index.tsx`：
   - 增加 `onMessage`
   - 初次加载按 URL 预设标题
3. 将 `xcx/src/pages/webview/index.config.ts` 初始标题改为通用标题
4. 在 H5 侧扩展 `miniProgramBridge.ts`：
   - 新增标题同步桥接方法
5. 在 H5 侧新增共享壳内上下文/标题同步工具
6. 在 `store/home.vue` 隐藏壳内顶部头部与吸顶头部
7. 在所有 `store/*` 页面接入共享标题同步函数
8. 验证支付回跳后的标题更新与页面布局
9. 更新文档说明“壳内标题由小程序原生导航栏承接，H5 头部在壳内隐藏”

- 从 `xcx` 启动页进入 H5 首页：
  - 外层显示 `店铺首页`
  - H5 首页不再出现重复顶部头部
- 从首页进入：

  - `xcx/pages/payment/index`
  - 完成/取消后回到 H5
  - 标题恢复正确

### 6.2 回归验证

  - 页面头部仍按原样展示
  - 不因壳内判断误伤普通 H5
- `openXcxPaymentPage()` 原有能力不受影响
- H5 登录态恢复、下单、订单详情、支付回跳链路不受影响

- 无新增类型错误/诊断错误

## 七、Risks & Notes

- `web-view` 消息桥依赖微信环境；因此标题动态同步需在微信开发者工具/真机内验证
- 如果某些 H5 页面不是通过 hash 路由进入，而是运行时局部切换内容，需要以实际路由变化点补发标题同步
- 当前“全部 H5 页面”在仓库里可确认的壳内范围主要是 `store/*`；若后续还要把商家后台放进 `web-view`，需要补一套 merchant 页面标题映射与头部隐藏规则
